# Skill — Go reproducible environment

**For**: generator.
**Applies when**: initializing or reviewing a Go project.

## The four files

```
go.mod              # module path + declared dependencies + Go version
go.sum              # hash-verified lockfile
.go-version         # exact Go release (e.g. 1.23.4)
Dockerfile          # multi-stage build using all three above
```

Missing any one of these is a reproducibility bug.

## go.mod

Minimal layout:

```
module github.com/<owner>/<repo>/service

go 1.23

require (
    github.com/gin-gonic/gin v1.10.0
    github.com/jackc/pgx/v5 v5.7.1
    github.com/golang-jwt/jwt/v5 v5.2.1
    github.com/cucumber/godog v0.15.0
    github.com/stretchr/testify v1.10.0
)
```

Rules:

- `go 1.23` is the **minimum** version the code uses. Don't
  inflate to 1.24 unless you use a 1.24-only feature.
- Pin by `MAJOR.MINOR.PATCH`; Go's modules system handles
  semver on top.
- `go mod tidy` before every commit. A `go.mod` with unused
  dependencies fails CI.

## go.sum

Generated by `go mod download` / `go mod tidy`. Check it in.
CI runs `go mod verify` to ensure the checked-in sums match
the actual module contents.

```bash
go mod tidy
go mod verify
```

## .go-version

```
1.23.4
```

One line, exact patch. Read by `asdf`, `gimme`, and the
project's CI config:

```yaml
# .github/workflows/ci.yml (excerpt)
- uses: actions/setup-go@v5
  with:
    go-version-file: .go-version
```

Update via:

```bash
echo 1.23.5 > .go-version
go mod edit -go=1.23
# commit both in the same commit
```

## golangci-lint

```yaml
# .golangci.yml
run:
  timeout: 5m
  go: "1.23"

linters:
  enable:
    - errcheck
    - gosimple
    - govet
    - ineffassign
    - staticcheck
    - unused
    - gofmt
    - goimports
    - misspell
    - errorlint
    - bodyclose
    - sqlclosecheck
    - rowserrcheck

issues:
  exclude-rules:
    - path: _test\.go
      linters:
        - errcheck  # t.Fatal in tests covers most err returns
```

Run locally: `golangci-lint run ./...` before commit. CI runs
the same. No PR merges with lint errors.

## docker compose

```yaml
# docker-compose.yml
services:
  api:
    build: .
    environment:
      - JWT_SECRET=${JWT_SECRET}
      - DATABASE_URL=postgres://api:api@db:5432/api?sslmode=disable
      - SERVICE_NAME=guardrail-api
    ports:
      - "8000:8000"
    depends_on:
      db:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "/api", "-healthcheck"]
      interval: 5s
      timeout: 3s
      retries: 12

  db:
    image: postgres:17-alpine
    environment:
      - POSTGRES_USER=api
      - POSTGRES_PASSWORD=api
      - POSTGRES_DB=api
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U api"]
      interval: 3s
      timeout: 2s
      retries: 20
```

`docker compose up -d --build` must bring both services to
`Up (healthy)` within ~15 seconds on a cold start. The `-healthcheck`
flag is a convention: the Go binary, when invoked with that
flag, makes an HTTP GET to `/health` on localhost and exits 0
or 1. Avoids the need for `wget` in the `distroless` image.

```go
// cmd/api/main.go — healthcheck snippet
if len(os.Args) > 1 && os.Args[1] == "-healthcheck" {
    resp, err := http.Get("http://localhost:8000/health")
    if err != nil || resp.StatusCode != http.StatusOK {
        os.Exit(1)
    }
    os.Exit(0)
}
```

## The `go run` dev flow

```bash
# Day-one:
go mod download

# Run with live-reload (air or equivalent):
air

# Or vanilla:
go run ./cmd/api

# Tests:
go test ./...
go test -run TestGuardrailService ./internal/services
go test -tags=integration ./tests/integration/...
```

## Cross-platform gotchas

- **`GOOS=linux GOARCH=amd64`** in the Dockerfile build
  step — guarantees the same binary regardless of where `docker
  build` runs (Apple Silicon, WSL2, x86 CI).
- **`//go:build`** tags for integration tests so
  `go test ./...` doesn't accidentally run them without their
  dependencies. Typical: `//go:build integration`.
- **CGO**: keep `CGO_ENABLED=0`. Adds SQLite support removal
  etc., but the `distroless/static` image has no libc; anything
  requiring CGO forces you to `distroless/base` (10× larger).

## What NOT to do

- **No vendor/ directory by default.** Go modules make it
  redundant. Projects that need offline build or strict
  supply-chain review can opt in (`go mod vendor`) and commit
  the tree; document in the project-level CLAUDE.md.
- **No `GOPATH` mode.** Modules only. A project without
  `go.mod` is pre-2019 legacy, not a new project.
- **No `latest` tag in `go.mod`.** `go get github.com/foo/bar`
  resolves to a concrete version at `tidy` time; rejected by
  `go mod verify` otherwise.
