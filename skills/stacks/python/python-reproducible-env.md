# Skill — Python reproducible environment (uv + lockfile)

**For**: generator.
**Applies when**: initializing or reviewing a Python project.

## The four files

Every Python project in this harness has, at minimum:

```
pyproject.toml     # declared dependencies + tool config
uv.lock            # resolved version pins
.python-version    # exact CPython version
Dockerfile         # container build using all three above
```

Missing any one of these is a reproducibility bug.

## pyproject.toml

`uv`-compatible minimal layout:

```toml
[project]
name = "guardrail-api"
version = "0.1.0"
requires-python = ">=3.13"
dependencies = [
    "fastapi>=0.115",
    "uvicorn[standard]>=0.32",
    "pydantic>=2.9",
    "pydantic-settings>=2.6",
    "sqlalchemy>=2.0",
    "asyncpg>=0.30",
    "pyjwt>=2.10",
    "httpx>=0.28",
]

[dependency-groups]
dev = [
    "pytest>=8.3",
    "pytest-asyncio>=0.24",
    "pytest-bdd>=7.3",
    "ruff>=0.8",
    "mypy>=1.13",
]

[tool.ruff]
line-length = 100
target-version = "py313"

[tool.ruff.lint]
select = ["E", "F", "I", "UP", "B", "SIM"]

[tool.pytest.ini_options]
asyncio_mode = "auto"
testpaths = ["tests"]
```

Rules:

- Pin minimum versions (`>=X.Y`) not exact (`==X.Y.Z`). The
  lockfile handles exactness.
- Separate dev deps via `[dependency-groups]`. Runtime image
  installs with `--no-dev`.
- Tooling config co-lives here (`[tool.ruff]`, `[tool.pytest]`).
  One source of truth; no `.flake8`, no `pytest.ini`.

## uv.lock

Generated by `uv lock` / `uv sync`. Check it in. CI runs `uv
sync --frozen` which fails if the lockfile disagrees with
`pyproject.toml`.

```bash
# First-time setup:
uv python install $(cat .python-version)
uv sync

# After changing deps:
uv add <pkg>            # updates pyproject + lockfile
uv remove <pkg>
uv lock --upgrade       # upgrade all pinned versions
```

Commit `uv.lock` and `pyproject.toml` together. The two are
always in sync; splitting them across commits is a merge-bomb
waiting to happen.

## .python-version

```
3.13.1
```

One line, the exact CPython version. `uv` reads it; `pyenv`
users read it; CI runners pin to it. `>=3.13` in pyproject is
the dependency contract; `3.13.1` in `.python-version` is the
reproducibility contract.

Update the file via `uv python pin 3.13.1`, never manually.

## The editable-install dev flow

```bash
# Day-one setup:
uv sync
source .venv/bin/activate
pytest                # runs against the local src/ live — no rebuild

# Or without activating:
uv run pytest
uv run uvicorn api.src.main:app --reload
```

`uv sync` installs the project in editable mode by default, so
edits to `api/src/*.py` are picked up without reinstall. This is
the local dev equivalent of `docker compose up --build`.

## docker compose for the service

```yaml
# docker-compose.yml
services:
  api:
    build: .
    environment:
      - JWT_SECRET=${JWT_SECRET}
      - DATABASE_URL=postgresql://api:api@db:5432/api
      - SERVICE_NAME=guardrail-api
    ports:
      - "8000:8000"
    depends_on:
      db:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"]
      interval: 5s
      timeout: 3s
      retries: 12

  db:
    image: postgres:17-alpine
    environment:
      - POSTGRES_USER=api
      - POSTGRES_PASSWORD=api
      - POSTGRES_DB=api
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U api"]
      interval: 3s
      timeout: 2s
      retries: 20
```

The compose file is how the evaluator's
`live-bdd-verification` skill exercises the service. `docker
compose ps` shows two services, both `Up (healthy)`.

## Cross-platform gotchas

- **`platform: linux/amd64`** on the `api` service if the team
  is mixed Apple Silicon / x86 and any wheel is
  architecture-specific. Uncommon but bites when it bites.
- **Line endings** — keep `.editorconfig` at repo root; do not
  rely on git autocrlf alone.
- **File watcher limits on Linux** — uvicorn `--reload` with
  many files may need `--reload-dir api/src`. Don't bump
  `fs.inotify.max_user_watches` in prod containers; bound the
  watched tree instead.

## What NOT to do

- `pip install -r requirements.txt` — no lockfile, no
  reproducibility contract. Use `uv`.
- `poetry install` — fine in isolation, but the harness
  standardizes on `uv` for speed. Projects that must use
  Poetry ship `poetry.lock` instead of `uv.lock` and document
  the divergence in the project-level CLAUDE.md.
- Conda / Anaconda — too heavyweight for service workloads. Use
  where the project is ML-research-first; document per-project.
- Multiple virtualenvs — one `.venv` per project, managed by
  `uv`. Devs who want isolation across branches use git
  worktrees, not parallel venvs.
