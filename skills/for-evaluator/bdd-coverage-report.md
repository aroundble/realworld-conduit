# Skill — Human-readable BDD coverage report

**For**: evaluator (authors + maintains), planner (reads for
Branch 4 prioritization), operator (reads for "what works?").
**Updated**: on every merge to the integration branch.

## Why this skill exists

The 2026-04-28 hot-deal retrospective had 83 merged PRs with CI
green, and the operator's first question was "so what can I
actually do?" There was no single artifact that answered that.
`summary.json` per run exists but is not human-readable.
`meta.json` per run has USER_INTENT but is per-run, not
per-feature.

This skill defines the **one file** that tells a human "here is
what users can do today, and with what evidence". It lives at:

```
tests/e2e/test-results/coverage-report.md
```

and is overwritten on every merge. It is the project's single
source of truth for "does this work for a real user".

## The report shape

```markdown
# BDD Coverage Report — <project>

**Last updated**: 2026-04-28T14:33:00Z (post-merge of PR #47)
**Deploy mode**: local-only
**Stack health at last update**: ✓ all services Up (healthy)

---

## User Behavior Tree

### Admin — can configure the platform
- ✓ **Admin registers a new guardrail and it takes effect immediately**
  — PR #47 · Issue #32 · scenarios: 1 · devices: 3/3 green
  — evidence: screenshots/32/admin-registers-guardrail.{desktop,mobile-chrome,webkit}.png
- ✓ **Admin cannot save a rule without severity**
  — PR #47 · Issue #32 · scenarios: 1 · devices: 3/3 green
- ✗ **Admin can delete an existing rule** — spec exists, last run failed on webkit
  — PR #49 (open, claim:evaluator) · devices: 2/3 green (webkit: timeout)
- – **Admin can bulk-import rules from CSV** — planned, issue #51 (claim:generator)

### End user — can browse the deal feed
- ✓ **User sees deals on landing page within 2s**
  — PR #12 · Issue #8 · scenarios: 1 · devices: 3/3 green
- ✓ **User filters by retailer**
  — PR #18 · Issue #14 · scenarios: 2 · devices: 3/3 green
- ✗ **User subscribes to a deal category** — spec exists, asserts fail on slow-3G
  — PR #24 (merged) · devices: 3/4 green (slow-3G: first-paint > 5s)
  — regression filed: Issue #53 (claim:generator)
- – **User sees comments on a deal detail page** — planned, issue #55

### End user — can compare prices
- – **User sees historical price chart** — planned, issue #58
- – **User sees cross-retailer comparison table** — planned, issue #59

---

## Legend

- **✓** — scenario green across all required devices, screenshot
  attached, merged.
- **✗** — scenario exists but currently red, OR merged without
  screenshot, OR device-matrix incomplete. Counts against
  integration-branch health.
- **–** — behavior named in roadmap but no spec yet. Counts as
  "planned", not broken.

---

## Counts

|  | ✓ | ✗ | – | Total |
|---|--:|--:|--:|--:|
| Admin | 2 | 1 | 1 | 4 |
| End user browses | 2 | 1 | 1 | 4 |
| End user compares | 0 | 0 | 2 | 2 |
| **Total** | **4** | **2** | **4** | **10** |

**Coverage ratio** (green / total): **40%**
**Implementation ratio** (green + red / total): **60%**
**Regression debt** (✗ count): **2**

---

## How to read this

- If "**✓** exists for the behavior you want to demo", it works
  at this commit. Check screenshot to confirm visual.
- If "**✗**", something regressed or was merged without complete
  evidence. Look at the linked issue/PR for the fix.
- If "**–**", the roadmap has it planned but no code. Planner's
  Branch 4 will pick the next one by priority.

---

*Generated by the evaluator on every merge. Do not edit
manually — edits are overwritten on the next merge.*
```

## The update procedure

On every `gh pr merge`, immediately:

```bash
# 1. Pull the updated integration branch.
git fetch origin && git checkout latest && git pull

# 2. Run the full BDD suite against fresh compose to get canonical
#    green/red status at the just-merged commit.
docker compose down -v
docker compose up -d --build
./scripts/wait-for-healthy.sh
E2E_ENV=local ./tests/e2e/run-e2e.sh --all-devices

# 3. Read the run results + issue bodies to update each behavior row.
#    The evaluator script is project-specific; generic shape below.

# 4. Write tests/e2e/test-results/coverage-report.md using the
#    template at top of this skill.

# 5. Commit + push to latest. Do NOT open a PR; this is a direct
#    commit by the evaluator on the integration branch — the
#    report IS the merge artifact.
git add tests/e2e/test-results/coverage-report.md
git commit -m "chore(coverage): update BDD coverage report after PR #<N>"
git push origin latest
```

The commit message ties the report update to the PR that
triggered it. `git log tests/e2e/test-results/coverage-report.md`
gives a timeline of user-visible capability changes.

## Composing the tree from issues + specs

Two-step composition:

**Step 1** — from **open + merged issues**, group by the `area/*`
label or the issue body's first-level heading. Each issue's title
becomes a behavior row. The issue's `claim:*` label decides its
status:
- `claim:evaluator` or merged → expect a green/red scenario
- `claim:generator` → the generator is working on it; scenario
  will exist soon
- `blocked` or `claim:human` → status `–` with reason appended
- No `claim:*` and not merged → `–` (planned)

**Step 2** — from **existing .feature files + latest Playwright/
godog run**, fill in:
- If the scenario ran green across the required device matrix
  AND screenshots are at the expected paths → `✓`
- If the scenario ran red on any required device, or screenshots
  are missing → `✗`
- If no matching scenario exists → `–`

The matching between behavior row and scenario is by **issue
number**. Feature files that say `# Issue #47` at the top, and
scenario titles that match the issue title, anchor the mapping.
See `skills/for-all-roles/bdd-acceptance-scenarios.md` §naming.

## Why this beats per-run summary.json

- **Stable URL.** `coverage-report.md` is always at the same
  path on the integration branch. An operator can bookmark it;
  a planner's Branch 4 knows where to read.
- **Human-readable.** Markdown renders on GitHub; no jq
  required.
- **User-behavior grouping.** Per-run reports group by test
  file. This report groups by "what users can do", which is
  the operator's mental model.
- **Regression debt is visible.** The ✗ count is a single number
  the operator watches; when it creeps up, the planner
  prioritizes fixes.
- **Historical diff.** `git log -p coverage-report.md` shows the
  addition/removal of each behavior over time.

## Integration with planner

Planner's Branch 4 reads `coverage-report.md` at every wake.
Priority rules:

1. Fix any ✗ in the top N most-important user flows before
   adding new – items. Regression debt compounds; new features
   on a broken base don't earn user value.
2. For remaining – items, convert the highest-priority rows
   into `claim:generator` issues with full BDD scenarios (per
   `skills/for-all-roles/bdd-acceptance-scenarios.md`).
3. If the ✓ ratio is under 30% **and** there are no open PRs
   in evaluator's queue, do not add more – items. The backlog
   is already too large for the team size; more breadth reduces
   demo-ability.

Planner does NOT edit `coverage-report.md` directly; it's an
evaluator-maintained artifact. Planner only reads.

## Integration with operator

Operator's "what can I demo?" answer is: open the report, look
for ✓, that's demoable. No need to read the repo, the test code,
or the issue tracker. One markdown file, always current.

Operator can also read the ✗ section as the short-term
quality-debt inbox; if anything there should not be broken, they
leave a comment on the linked issue to bump priority.

## When the report doesn't apply

- Pure library / CLI projects with no user behavior — skip
  this skill; use `README.md` + example scripts instead.
- Pure backend / internal-tool projects where there is no
  end-user — organize the tree by "integration partner" or
  "internal dashboard" instead of "end user". The skill shape
  is the same; only the role names in headings change.
