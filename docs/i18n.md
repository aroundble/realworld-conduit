# Internationalization (#167a — walking skeleton)

Conduit supports five UI-chrome locales: English (default), Korean, Japanese, Spanish, German. Current scope (#167a) covers the highest-visibility chrome surfaces — Navbar, Footer, SkipLink, ErrorState, root NotFound, KeyboardShortcutHelp + the locale switcher. Article / comment / editor / auth / settings surfaces remain English and are tracked in per-surface follow-up issues #167b–e. SEO hreflang alternates (sitemap, JSON-LD, OG, per-route metadata) land last in #167e.

## URL routing

`localePrefix: "as-needed"`:
- `/` + `/article/<slug>` stay un-prefixed for the default locale (English).
- `/ko`, `/ja`, `/es`, `/de` + children (`/ko/article/<slug>`) carry the locale prefix.

The middleware (`apps/web/middleware.ts`) runs before every request and:
1. Mints / forwards `X-Request-ID` for observability (#25 floor).
2. Resolves the locale via next-intl's `createMiddleware`:
   - `conduit-locale` cookie wins if set.
   - Otherwise, `Accept-Language` drives a first-visit 307 redirect to the best-matching locale prefix.
3. Stamps `x-request-id` onto the response headers.

The app router tree lives at `src/app/[locale]/` so the `[locale]` segment appears as a Next.js dynamic param. The root `src/app/layout.tsx` stays locale-agnostic (HTML + body + providers); the per-locale layout at `src/app/[locale]/layout.tsx` validates the locale + calls `setRequestLocale` so deeper server components can use `getTranslations()` without threading the locale.

## Message bundles

All five bundles live at `apps/web/messages/<locale>/common.json` with matching key sets. `en/common.json` is the source of truth; ko/ja/es/de were AI-translated as bootstrap for the walking skeleton (Navbar, Footer, SkipLink, shortcut modal, ErrorState, NotFound — 30 keys × 5 locales = 150 translation entries). Native-speaker review is expected per-surface as follow-up PRs #167b–e ship.

The key set is frozen by a CI gate — `pnpm i18n:check` diffs every locale against `en` and exits 1 on divergence. A developer adding a new key only to `en/common.json` breaks the check, forcing them to provide the matching key for every locale (even if temporarily with the English value).

## Locale switcher

`apps/web/src/components/LocaleSwitcher.tsx` renders a native `<select>` in the Navbar. Changing the option:
1. Writes `conduit-locale=<next>` cookie (Max-Age 1 year, SameSite=Lax).
2. Navigates via `window.location.href = "/<next-prefix>/<pathname>"` — a full-document navigation is required because next-intl resolves the locale once per server render; a `router.replace` would leave the hydrated client components reading the stale locale.

Native `<select>` intentionally — zero custom JS, works with keyboard + screen readers for free, and passes axe without a combobox ARIA pattern.

## Viewer detection in client components

Client components that need the active locale use `useLocale()` from next-intl. Server components use `const t = await getTranslations("ns")` (async call on every render). The root layout wraps everything in `<NextIntlClientProvider locale={locale} messages={messages}>` so the bundle flows to client components.

## Accept-Language redirect behavior

next-intl 4's middleware redirects on `Accept-Language` but does NOT set the `conduit-locale` cookie on the redirect itself. The cookie is written only on an explicit switcher action. Rationale: if the cookie were set on auto-redirect, a user with a preferred language that doesn't yet have all strings translated would be locked in and never see the alternate-locale UI they might prefer. Subsequent user-initiated navigations + switcher use set the cookie and persist the preference.

## Adding a new UI string

1. Add the key to `apps/web/messages/en/common.json` under the appropriate namespace (`nav`, `footer`, `shortcuts`, etc.).
2. Add the **same key** to `ko/common.json`, `ja/common.json`, `es/common.json`, `de/common.json` — with translated values, or the English value as a bootstrap if you don't have translations yet. `pnpm i18n:check` will fail the CI gate otherwise.
3. Reference the key in code via `useTranslations("namespace")` (client) or `getTranslations("namespace")` (server).
4. If the string is in a component outside the walking-skeleton chrome, follow the per-surface follow-up issue pattern (#167b–e) — don't convert everything to `t()` in one sweep.

## Adding a new locale

1. Add the ISO code to `routing.locales` in `apps/web/src/i18n/routing.ts`.
2. Create `apps/web/messages/<code>/common.json` with the same key set as `en/common.json`.
3. Add a label entry to `LocaleSwitcher.tsx`'s `LABELS` map (native script preferred, e.g. "Français" not "French").
4. Verify: `pnpm i18n:check && pnpm test:e2e tests/e2e/specs/167a-web-i18n-walking-skeleton.spec.ts`.
